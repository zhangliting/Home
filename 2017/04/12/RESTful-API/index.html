RESTful API

Posted on April 12, 2017

#RESTful API

  • 本质：一种软件架构
  • 核心：面向资源
  • 解决的问题：
    • 降低开发的复杂性
    • 提高系统的伸缩性

设计概念和准则

  • 网络上的所有事物都可以被抽象为资源。
  • 一个资源都有唯一的资源标识，对资源的操作不会改变这个标识。
  • 所有的操作都是无状态的。
  • RESTful对于资源型服务接口来说很合适，同事特别适合对于效率要求很高，但是对于安全要求不高的场景

HTTP协议 - UR

schema://host[:port]/path?[?query-string][#anchor]

HTTP协议 - 请求

组成格式：请求行、消息报头、请求正文
请求行
格式：Method Request-URI HTTP-Version CRLF
举例：GET/HTTP/1.1 CRLF
请求方法

  • Get 请求获取Request-URI所标识的资源
  • POST 在Request-URI所标识的资源后附加新的数据
  • HEAD 请求获取由Request-URI所标识的资源的响应消息报头
  • PUT 请求服务器存储一个资源，并用Request-URI所为其标识
  • DELETE 请求服务器删除Request-URI所标识的资源
  • OPTIONS 请求查询服务器的性能，或者查询与资源相关的选项和需求

HTTP协议 - 响应

组成格式：状态行、消息报头、响应正文
状态行
格式：HTTP-Version Status-Code Reason-Phrase CRLF
举例：HTTP/1.1 200 OK
状态码

  • 200 ok //客户端请求成功
  • 400 Bad Request //客户端请求有语法错误，不能被服务器所理解
  • 401 Unauthorized //服务器收到请求，但是拒绝提供服务
  • 404 Not Found //请求资源不存在
  • 500 Internal Server Error //服务器发生不可预期的错误
  • 503 Server Unavailable //服务器当前不能处理客户端的请求

SOAP WebService

WebService是一种夸编程语言和夸操作系统平台的远程调用技术。
WebService通过HTTP协议发送请求和接受结果时采用XML格式封装，并增加了一些特定的HTTP消息头，这些特定的HTTP消息头和XML内容格式就是SOAP协议。

如何设计RESTful API

资源路径（URI）

在RESTful架构中，每个地址代表一种资源，所以网址中不能有动词，只能名词。一般用复数形式。

HTTP动词

对于资源的操作(CURD)，有HTTP动词（谓词）表示。

  • GET
  • POST
  • PUT
  • PATCH
  • DELETE

过滤信息

如果记录数量很多，API应该提供参数，过滤返回结果。

状态码

服务器想用户返回状态码和提示信息，使用标准HTTP状态码。

错误处理

如果状态码是4XX或者5XX，就应该想用户返回错误信息
{“error”:”参数错误”}

返回结果

  • GET/collections:返回资源对象的数组
  • GET/collections/indentity:返回单个资源对象
  • POST/collections:返回新生成的资源对象
  • PUT/collections/indentity:返回完整的资源对象
  • PATCH/collections/indentity:返回被修改的属性
  • DELETE/collections/identity:返回一个空文档
